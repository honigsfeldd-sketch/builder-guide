01 · בסיס

איך מוצר דיגיטלי עובד

מוצר דיגיטלי הוא כלי שעוזר למשתמש לעשות משהו טוב יותר, מהר יותר או פשוט יותר.

התקדמות קריאה 0%

מסך

נקודת מפגש אחת בין המשתמש למוצר. לדוגמה: דשבורד, הרשמה או עמוד תשלום.

Feature

יכולת אחת בתוך המוצר. לדוגמה: כפתור “שלח תזכורת”.

MVP

הגרסה הכי קטנה שאפשר לבדוק עם משתמשים אמיתיים.

מושג הסבר קצר דוגמה
משתמש האדם שעבורו בונים את המוצר. פרילנסר שרוצה לעקוב אחרי לקוחות.
בעיה משהו שמפריע למשתמש. קשה לדעת מי שילם ומי לא.
User Flow הדרך שהמשתמש עובר במוצר. הרשמה → יצירת לקוח → שליחת חשבונית.
Prototype דגם ראשוני לבדיקה לפני בנייה מלאה. מסכים לחיצים ב-Figma.
02 · חשיבה מוצרית

לפני שבונים, שואלים

לא מתחילים מהפתרון. מתחילים מהבעיה, מהמשתמש ומהבדיקה הכי קטנה שאפשר לעשות.

לא “כולם”. אדם ברור עם צורך ברור. למשל: בעלת עסק קטן שמנהלת פגישות לבד.
הכאב הוא לא רעיון לפיצ׳ר. הוא החיכוך בחיים של המשתמש: זמן, בלבול, כסף, טעויות.
הגרסה הכי קטנה שמוכיחה שיש ערך. כל מה שלא חייבים עכשיו עובר לגרסה הבאה.
1

Discover

חוקרים את המשתמש, הבעיה והמצב הקיים.

2

Define

מגדירים בדיוק איזו בעיה פותרים.

3

Develop

מפתחים כמה רעיונות לפתרון.

4

Deliver

בונים גרסה קטנה, בודקים ומשפרים.

03 · UX/UI

ממשק טוב מוריד מאמץ

UI הוא איך זה נראה. UX הוא איך זה עובד ומרגיש. יחד הם צריכים להיות ברורים, צפויים ונוחים.

היררכיה

מה רואים קודם, ומה פחות חשוב. מסך טוב לא נותן לכל דבר לצעוק.

ריווחים

המרחקים בין אלמנטים. ריווח נכון יוצר תחושת סדר ונשימה.

טיפוגרפיה

גודל, משקל וסגנון הטקסט. טקסט ברור הוא חלק מהמוצר, לא קישוט.

CTAהפעולה המרכזית במסך. למשל: “המשך”.
Loadingמה קורה כשהמערכת טוענת מידע.
Errorמה המשתמש רואה כשמשהו נכשל.
Emptyמה מוצג כשאין עדיין תוכן.
04 · טכני

להבין תפקיד, לא לשנן קוד

המטרה היא לדעת מה כל חלק עושה במוצר כדי לדבר ברור עם AI, מפתחים וכלי בנייה.

Frontend הוא מה שהמשתמש רואה. Backend הוא הלוגיקה מאחורי הקלעים. Database הוא המקום שבו שומרים מידע.
API הוא דרך שבה מערכות מדברות. Integration הוא חיבור בין המוצר שלך לשירות אחר, כמו סליקה או אימייל.
Hosting הוא המקום שבו המוצר רץ. Deploy הוא העלאה לאוויר. Domain הוא הכתובת שאנשים מקלידים.
GitHub שומר ומנהל קוד. Environment Variables הן הגדרות סודיות כמו מפתחות API וסיסמאות.
05 · AI

AI עובד טוב עם מסגרת

ככל שהמשימה קטנה וברורה יותר, כך התוצאה טובה יותר. נותנים הקשר, מגבלות ויעד אחד בכל פעם.

Prompt

ההוראה שנותנים ל-AI. היא צריכה לכלול מטרה, הקשר ותוצאה רצויה.

Context

הרקע שה-AI צריך לדעת: מי המשתמש, מה כבר קיים ומה מגבלות המשימה.

Scope

גבולות המשימה. מה בפנים, מה בחוץ ומה דוחים לשלב הבא.

פרומפט חלש

תבנה לי אפליקציה לניהול כסף.

רחב מדי. אין משתמש, אין מסכים, אין גבולות.

פרומפט טוב

אני בונה MVP לפרילנסרים: להוסיף הכנסה, להוסיף הוצאה ולראות סיכום חודשי. הצע User Flow ושלושה מסכים ראשונים. בלי פיצ׳רים מתקדמים.

יש משתמש, מטרה, גבולות ותוצר ברור.

06 · אחריות

מוצר טוב מכבד משתמשים

גם מוצר קטן משפיע על אנשים. אמון הוא לא שכבת עיצוב, הוא החלטה מוצרית.

פרטיות

לא אוספים דאטה שלא באמת צריך, ולא מסתירים מה עושים איתו.

בלי Dark Patterns

לא דוחפים משתמש לפעולה שמשרתת את המוצר אבל פוגעת בו.

שקיפות

מידע חשוב צריך להיות ברור בזמן הנכון, לא קבור באותיות קטנות.

נגישות

ממשק צריך להיות שימושי גם למי שקורא לאט, מנווט אחרת או צריך יותר בהירות.

07 · תהליך

סדר עבודה פשוט

לא בונים הכול ביום הראשון. בונים משהו קטן שאפשר ללמוד ממנו.

להבין בעיהלנסח מה באמת מפריע למשתמש.
להגדיר משתמשלבחור למי בונים קודם.
לכתוב User Flowלשרטט את הדרך המרכזית במוצר.
לשרטט מסכיםלהפוך את הזרימה למסכים פשוטים.
להחליט MVPלהשאיר רק מה שחייבים כדי לבדוק ערך.
לבנות Prototypeליצור דגם ראשוני שאפשר להראות.
לבדוק עם אנשיםלקבל תגובה אמיתית, לא רק תחושת בטן.
לתקן לפי פידבקלשפר את מה שמבלבל או לא עובד.
להעלות לאווירלתת למשתמשים אמיתיים לנסות.
לשפר בהדרגהלהתקדם לפי למידה, לא לפי רעש.
